Overview

Orbit OS is an open-source distributed task scheduler built with Go and etcd. It provides reliable, at-least-once task execution across a cluster of worker nodes with automatic leader election, worker failover, and a real-time web dashboard.

It was born out of frustration with existing schedulers that were either too heavyweight (Airflow, Temporal) or too primitive (cron). Orbit occupies the middle ground: simple to operate, production-grade in its guarantees.

The Problem

"We need something between cron and Airflow. Something that handles node failures gracefully but doesn't require a PhD to operate."

— Common feedback from the distributed systems community

Most teams running background jobs face the same tradeoff: simple schedulers lack fault tolerance, and fault-tolerant schedulers are operationally complex. Orbit provides consensus-backed scheduling with a single binary and a 5-minute setup.

Architecture

Orbit uses etcd for distributed state and leader election. The leader node runs the scheduler loop, distributing tasks to worker nodes via gRPC. If the leader fails, etcd's lease expiry triggers a new election within 10 seconds. Workers heartbeat their task progress; if a worker disappears mid-task, the task is re-queued automatically.

The scheduler uses a weighted priority queue. Tasks carry a priority, a cron expression or one-shot trigger, a retry policy, and an optional dependency list for DAG-style workflows.

Key Engineering Decisions

01

etcd over Raft-from-scratch

Building a Raft implementation would have been academically interesting but operationally risky. etcd is battle-tested at Kubernetes scale. Using it as the consensus layer let the project focus on scheduling semantics rather than distributed systems primitives.

02

gRPC for worker communication

gRPC's bidirectional streaming is ideal for heartbeating. Workers open a persistent stream to the leader, sending progress ticks every 2 seconds. No polling, no HTTP overhead, and the stream itself acts as a liveness signal.

03

Plugin-based task execution

Task executors are Go plugins loaded at startup. This lets users add custom executors (HTTP, shell, Docker, Kubernetes Job) without forking the project. The community has contributed 14 executor plugins.

Lessons Learned

Maintaining open source is a product discipline, not just an engineering one. The biggest growth lever wasn't features — it was documentation. After rewriting the getting-started guide with a 5-minute Docker Compose demo, weekly installs tripled within a month.

Issue triage is a full-time job above 500 stars. I introduced a good first issue / help wanted labelling system and a CONTRIBUTING guide that walks through the codebase. First-time contributions went from 2/month to 12/month.
