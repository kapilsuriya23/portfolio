Overview

Job Application Tracker is a full-stack web application built to eliminate the chaos of tracking job applications across spreadsheets, browser tabs, and sticky notes. It centralises the entire job-hunt workflow — from first application to offer — into a single authenticated dashboard with a visual Kanban pipeline, rich analytics, and proactive email reminders.

The project was designed specifically with placement season in mind: fast to set up, zero vendor lock-in, and fully self-hostable. The frontend is deployed on Vercel and the Express API on Render, with MongoDB Atlas as the cloud database.

The Problem

"I missed two interview deadlines last semester because I forgot I had applied. I was tracking everything in a Google Sheet and it just didn't scale."

— Feedback from a peer during ideation

Students applying for jobs typically manage 20–50 applications simultaneously across different companies, timelines, and stages. A static spreadsheet has no status pipeline, no visual overview, and no ability to send proactive alerts — leading to missed deadlines and lost opportunities.

Architecture

The application follows a standard SPA + REST API architecture, with the frontend and backend deployed independently to keep concerns separated and allow each layer to scale on its own.

01

Kanban Drag-and-Drop Board

Built with @dnd-kit, the Kanban board lets users drag application cards across all 6 pipeline stages. The drag overlay is rendered via a React Portal to avoid z-index stacking context issues across Kanban columns — a non-trivial fix that required understanding how the DOM paint order works with position: fixed.

02

Analytics Dashboard with Recharts

Four interactive charts built with Recharts: application trends over time (LineChart), status distribution (PieChart), pipeline funnel (BarChart), and success rate metrics. All charts are responsive and update in real time as application data changes.

03

Automated Email Reminders

A node-cron job scheduled at midnight queries for upcoming deadlines and fires styled HTML emails through Brevo SMTP. Gmail App Passwords were unavailable on the institutional account, making Brevo the pragmatic SMTP choice — it also provides better deliverability and a generous free tier.

04

JWT Authentication & Security

Stateless JWT-based auth with bcrypt password hashing, rate limiting on auth endpoints, and CORS configured to allowlist both the Vercel production domain and Vercel preview URLs — a configuration subtlety that tripped up early deployments and required dynamic origin validation.

05

Vite Bundle Optimisation

Manual chunk splitting in vite.config.js separates recharts, dnd-kit, and react-vendor into independent chunks. This improves initial load performance by allowing the browser to cache heavy dependencies independently from application code that changes frequently.

Technical Challenges

The three hardest problems were CORS on Vercel preview deployments, the drag overlay z-index bug, and Brevo SMTP setup — all solved in production.

— Engineering notes

CORS on Vercel previews: Each Vercel preview URL is unique (interntrack-git-feat-xxx.vercel.app). A static allowlist breaks every PR preview. The fix was a dynamic origin validator on the Express server that uses a regex to match the Vercel preview URL pattern alongside the production domain.

Drag overlay with @dnd-kit: Rendering the drag ghost inside a Kanban column caused it to be clipped by overflow: hidden on the column container. Moving the DragOverlay to a React Portal at the document body level resolved the clipping without any CSS hacks.

Brevo SMTP: Institutional Google accounts block App Passwords, making Nodemailer + Gmail the standard go-to infeasible. Brevo's SMTP relay works with any Nodemailer transport config and adds open/click tracking as a bonus.

Lessons Learned

Deployment environment differences are not edge cases — they're the rule. CORS, environment variables, and preview URL patterns all behave differently in production than in local development. Building the CORS validator to handle dynamic origins from day one would have saved two debugging sessions.

The Vite chunk splitting exercise was valuable beyond performance: understanding which libraries change rarely (Recharts, dnd-kit) vs. frequently (app code) is a transferable mental model for any bundler or CDN caching strategy.

For a future version, I'd add WebSocket-based real-time updates so collaborators (e.g. a placement cell coordinator) can see application status changes live without a page refresh.
