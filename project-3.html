Overview

Synapse is a GitHub App and Actions integration that performs AI-assisted code review on every pull request. It parses the diff, understands the surrounding codebase context, and leaves structured review comments covering bugs, security issues, performance concerns, and style violations — in under 45 seconds.

It's designed to complement, not replace, human review: Synapse handles the mechanical first pass so engineers can focus their review time on architecture and intent.

The Problem

"We have a 48-hour average time-to-review. Junior engineers wait two days to find out they used the wrong error handling pattern. It's killing our velocity."

— Engineering Manager, client team

Code review bottlenecks are one of the most consistent friction points in engineering teams. Reviewers are busy, context-switching is expensive, and mechanical issues (null checks, missing tests, security anti-patterns) consume review time that should be spent on higher-level discussion.

Architecture

Synapse runs as a GitHub App triggered by the pull_request webhook. A Next.js API route receives the event, fetches the diff and surrounding file context via the GitHub REST API, and passes a structured prompt to GPT-4o. The response is parsed into structured review comments and posted back via the GitHub Pull Request Review API.

Key Engineering Decisions

01

Structured outputs over free-form text

Using OpenAI's response_format: json_schema, every review is returned as a typed array of comment objects with file, line, severity, and body fields. Zero regex parsing, zero hallucinated line numbers.

02

Context windowing for large PRs

Large diffs exceed the context window. Synapse chunks the diff by file, processes each independently, then deduplicates and ranks comments by severity before posting. PRs with 50+ files are handled gracefully.

03

Team-specific rule injection

A .synapse.yml config file lets teams define custom rules in plain English ("Always check for SQL injection in database query functions"). These are injected into the system prompt, making reviews team-aware without fine-tuning.

Lessons Learned

False positives were the biggest adoption risk. Early versions flagged too many style issues that teams didn't care about, and engineers started ignoring all comments. I added a feedback thumbs-up/down on each comment that feeds into a per-team suppression model — after one week, false positive rate drops by ~60%.

The system prompt matters as much as the model. Spending two days on prompt engineering reduced hallucinated line numbers from 12% to under 1% — a bigger win than switching model versions.
