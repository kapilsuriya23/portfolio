Overview

AlgoLens is a browser extension built for students who get stuck on DSA problems and reach for the solution too quickly. Instead of dumping the answer, it walks the student through five progressive steps — plain-English explanation, algorithmic pattern, step-by-step approach, ordered hints, and finally a solution that is locked behind an explicit user action.

The extension auto-detects the problem you're viewing, extracts its content via platform-specific strategies, calls an Express backend that queries OpenRouter AI, and renders a structured analysis in a popup — all without leaving the problem page. Supported platforms: LeetCode, GeeksForGeeks, HackerRank, and Codeforces.

The Problem

"I open the editorial after 5 minutes, read the solution, feel like I understood it — and then fail the exact same problem type next week."

— The cycle AlgoLens is designed to break

The standard student workflow on LeetCode is all-or-nothing: struggle briefly, open the full editorial, passively read the answer. This produces the illusion of understanding without the retention that comes from productive struggle. AlgoLens replaces the editorial with a graduated reveal that requires active engagement at each step before unlocking the next.

Architecture

A Chrome extension runs in three isolated JavaScript environments that cannot share memory — they communicate exclusively through Chrome's messaging APIs. Understanding this constraint shaped every architectural decision.

Key Engineering Decisions

01

GraphQL API over DOM scraping for LeetCode

LeetCode's class names change frequently with frontend deployments, making CSS selectors brittle. By calling their public GraphQL endpoint with the problem slug extracted from the URL, title and description extraction is completely immune to DOM changes — the most reliable extraction strategy across all four platforms.

02

Manifest V3 service worker constraints

MV3 replaces persistent background pages with event-driven service workers that terminate when idle. All async message handlers required explicit return true to keep the Chrome message channel open, and incoming messages needed to handle the case where the service worker had been terminated and required re-awakening.

03

Structured JSON prompt as a typed API contract

The AI prompt specifies the exact output schema field by field, instructs the model to return only raw JSON with no markdown wrapping, and sets temperature to 0.2 for consistency. This turns the LLM call into something closer to a typed API — the backend JSON.parse()s directly and validates field presence before forwarding. Defensive stripping of markdown code fences handles the edge cases where the model wraps output in triple backticks despite instructions.

04

Retry + caching in the content script

LeetCode, GFG, and HackerRank all render content via React, so the DOM isn't ready when the content script first runs. Extraction retries up to 12 times at 700ms intervals. The result is cached in the content script's module scope, so subsequent popup opens don't re-trigger scraping. A MutationObserver handles SPA navigation when users switch problems without a full page reload.

05

Cross-platform DOM extraction strategy

Four platforms, zero shared HTML structure, and GFG has two distinct page layouts. Solved with a priority-ordered selector list, <title> tag fallback, and a heuristic largest-text-block scan as a last resort — with garbage detection to filter out navigation and banner text that DOM scanners otherwise pick up first.

Challenges & Fixes

⚡

ESM compatibility in Vite config

Vite configs run as ESM when "type": "module" is set in package.json, which means __dirname doesn't exist. Fixed by reconstructing it with fileURLToPath(import.meta.url) and dirname() — a subtle Node.js ESM gotcha that breaks the build silently if missed.

⚡

LLM output consistency

Models occasionally wrap JSON in markdown fences, include trailing commas, or truncate long outputs. Added defensive cleaning (strip fences before parsing, set max_tokens: 3000) to make parsing reliable regardless of which model OpenRouter routes to.
