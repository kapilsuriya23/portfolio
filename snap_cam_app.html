Overview

VaultPay is a developer-first payment SDK that abstracts away the complexity of PCI-DSS compliance, multi-provider routing, and webhook handling behind a clean Python API. It was built after noticing that every fintech startup I worked with spent 3–6 months on payment infrastructure before writing their first line of product code.

With VaultPay, a developer can accept payments from 15+ methods — cards, ACH, SEPA, wallets — with fewer than 30 lines of code, fully compliant on day one.

The Problem

"We spent our entire first sprint just getting Stripe and PayPal talking to each other. We hadn't written a single feature for our actual users."

— Founder, early-stage fintech client

Payment infrastructure is notoriously repetitive: idempotency keys, webhook signature verification, retry logic, failed payment recovery, provider failover. Every startup reimplements these from scratch, often incorrectly.

Architecture

VaultPay is structured as two layers: a thin SDK (pip-installable) that handles the developer-facing API, and a serverless gateway hosted on AWS that manages provider routing, tokenization, and compliance controls.

Key Engineering Decisions

01

Idempotency by default

Every mutating API call requires an idempotency key, stored in DynamoDB with a 24-hour TTL. Duplicate requests return the cached response without re-charging the customer — a bug that costs fintech startups thousands monthly.

02

Provider-agnostic routing

An internal routing layer selects the best provider per transaction based on currency, method, and client-defined cost rules. Failover is automatic — if Stripe returns a 5xx, the transaction retries via Adyen within 800ms.

03

HMAC-signed webhooks with replay protection

Every outgoing webhook is signed with the client's secret and includes a timestamp. The SDK's receiver validates both the signature and that the timestamp is within a 5-minute window, preventing replay attacks.

Lessons Learned

Testing payment flows is uniquely painful — Stripe's test mode is good but doesn't cover every edge case (e.g. bank-side declines, network timeouts mid-auth). I built a deterministic simulation layer that injects failure modes on demand during integration tests, which caught 6 bugs before production.

SDK versioning discipline matters more than I anticipated. After the first breaking change, I adopted a strict semantic versioning policy with a deprecation window, which saved several client relationships.
